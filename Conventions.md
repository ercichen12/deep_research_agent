# Conventions

## Project Shape

This is a Next.js app with TypeScript, React, Vitest, and file-based research persistence.

Top-level folders:

- `app/` - Next.js App Router pages, styles, and API routes.
- `lib/` - research engines, provider integrations, storage, model calls, and shared utilities.
- `lib/heavy/` - Heavy research domain modules.
- `tests/` - Vitest tests for API routes, Heavy modules, UI, and legacy research utilities.
- `docs/superpowers/specs/` - design documents.
- `docs/superpowers/plans/` - implementation plans.
- `research-runs/` - local runtime artifacts generated by Heavy inquiries.
- `web-articles/` - local fetched article/cache artifacts. 待确认: exact current producer and retention policy.

## Documentation

The three root project logs are:

- `README.md` - practical project overview and run instructions.
- `Decisions.md` - dated product and technical decisions.
- `Conventions.md` - repository and workflow conventions.

Designs and plans go under `docs/superpowers/` with date-prefixed filenames:

```text
docs/superpowers/specs/YYYY-MM-DD-topic-design.md
docs/superpowers/plans/YYYY-MM-DD-topic.md
```

## Heavy Module Conventions

Current Heavy modules live under `lib/heavy/`.

Existing legacy engine responsibilities:

- `types.ts` owns shared types, defaults, JSON parsing, and normalizers.
- `storage.ts` owns file storage and event-log IO.
- `events.ts` owns event helper behavior.
- `search-provider.ts` owns relay/OpenCLI/web/fetch search policy and provider traces.
- `coordinator.ts` owns legacy run-level task planning.
- `agent-runner.ts` owns concurrent task execution.
- `adaptive-research.ts` owns per-agent adaptive search/read steps.
- `verifier.ts` owns legacy run-level evidence-gap checks.
- `synthesizer.ts` owns legacy final report synthesis.
- `orchestrator.ts` owns the legacy Heavy run loop.

Planned Graph Heavy modules should be placed under:

```text
lib/heavy/graph/
```

and should keep the same separation of concerns:

- frame creation
- state persistence/update
- action planning
- action execution
- evidence extraction
- candidate pool merge/promotion
- evaluation/replanning
- ranking
- finalization

## Runtime Storage Conventions

Generated runtime files should stay under `research-runs/`.

Current paths:

```text
research-runs/inquiries/{inquiryId}.json
research-runs/logs/{turnId}.ndjson
research-runs/sources/{sourceHash}.json
```

Planned graph-state path:

```text
research-runs/graph-state/{turnId}.json
```

Event logs must use one JSON object per line. They must not include API keys or full raw provider error bodies.

## Search And Source Logging

Search behavior should be visible and auditable:

- Log provider name.
- Log engine name when available, such as `google`, `brave`, `duckduckgo`, `bing`, `relay`, or `web`.
- Log query text.
- Log result count and returned result summaries.
- Log page reads and read status.
- Store captured sources with title, URL, snippet/full text when available, provider, and engine.

Search queries should be English by default. Do not include internal task IDs, role names, or literal placeholders such as `"CEO name"`, `"company name"`, or `"{company}"`.

## Security Conventions

- Do not commit `.env.local`.
- Do not write API keys to docs, tests, logs, snapshots, or UI.
- Health routes may report provider status but must not echo secrets.
- Error messages should be compact and secret-redacted.

## UI Conventions

The app should render the actual working console first, not a landing page.

Heavy UI should display real data from inquiry JSON and event logs:

- inquiry status
- research stages
- run/cycle process
- agent/action logs
- provider and engine traces
- captured sources
- verifier/evaluator output
- final Markdown

For planned Graph Heavy, add visible panels for:

- Research Frame
- Research Process timeline
- Candidate Pool
- Evidence Matrix
- Rejected Paths
